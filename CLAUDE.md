# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Blast Radius Code Analyzer** — a CLI tool that:
1. Parses `git diff` to find changed functions
2. Traverses a FalkorDB code graph (via CodeGraphContext) to find all upstream callers
3. Sends the subgraph to Gemini to synthesize targeted pytest tests
4. Runs those tests and logs every interaction to SQLite for future fine-tuning

## Environment Setup

```bash
# Create venv and install deps
python -m venv .venv && source .venv/bin/activate
pip install -e .

# Pin this — v1.x installs as _native module and breaks CGC
pip install tree-sitter-language-pack==0.6.0

# Index a target repo before running the pipeline
cgc index /path/to/target/repo

# Verify the FalkorDB graph is populated
cgc analyze callers <any_function_name>
```

Copy `.env.example` to `.env` and fill in:
```
FALKORDB_SOCKET=/root/.codegraphcontext/global/db/falkordb.sock
GEMINI_PROJECT=your-gcp-project
GEMINI_LOCATION=us-central1
CGC_GRAPH_NAME=codegraph
```

## Running the Tool

```bash
# Full pipeline: diff → blast radius → synthesize tests → run tests
python -m blast_radius /path/to/target/repo

# Dry run: print blast radius without synthesizing/running tests
python -m blast_radius /path/to/target/repo --dry-run
```

## Running Tests

```bash
pytest tests/ -v
pytest tests/test_graph.py -v          # graph client tests only
pytest tests/ -v --tb=short            # short tracebacks
```

## Architecture

### Data Flow

```
git diff --unified=0 HEAD
    │
    ▼
ChangedRange(file_path, [line_numbers])        ← git_diff.py
    │
    ▼  graph lookup: line_number <= L <= end_line
FunctionNode (ground zero)                     ← resolver.py
    │
    ▼  Cypher: MATCH (caller)-[:CALLS*1..10]->(changed)
BlastRadiusResult                              ← graph.py
    │
    ├── affected_functions (deduplicated)
    ├── entry_points (no callers above them)
    └── call_chains (human-readable paths)
    │
    ▼  build_context() → Gemini → test code
pytest file written + executed                 ← synthesizer.py + runner.py
    │
    ▼
interactions.db row (prompt, output, pass/fail) ← telemetry.py
```

### FalkorDB / CGC Graph Schema

```
Nodes:  Repository, File, Function, Class, Variable, Parameter, Module
Edges:  CONTAINS, IMPORTS, HAS_PARAMETER, CALLS

Function properties: name, source, line_number, end_line, cyclomatic_complexity, decorators, docstring
CALLS properties:    line_number, full_call_name
```

Key insight: `source` is stored on every `Function` node — no filesystem reads needed to build the Gemini context.

### Core Cypher Queries (in `graph.py`)

- **Find by name:** `MATCH (f:Function {name: $name})`
- **Find by file + line:** `WHERE file.path=$path AND f.line_number <= $line AND f.end_line >= $line`
- **Blast radius:** `MATCH p=(entry)-[:CALLS*1..10]->(changed:Function {name: $fn_name})`
- **Entry points only:** add `WHERE NOT ()-[:CALLS]->(entry)` to blast radius query

FalkorDB handles cycles in variable-length paths natively — no explicit visited-set needed.

### Telemetry / Training Loop

Each `interactions` row is a complete supervised fine-tuning example:
- **Input:** `prompt` (the subgraph context sent to Gemini)
- **Output:** `final_code` if developer edited it (`edited=1`), else `generated`
- **Quality signal:** `passed / (passed + failed)` ratio

## Known Pitfalls

| Issue | Fix |
|---|---|
| `tree-sitter-language-pack` breaks CGC | Pin to `==0.6.0` |
| CGC index is stale when diff runs | Re-run `cgc index` at pipeline start, or use `cgc watch` |
| Gemini returns markdown fences | Strip fences and validate with `ast.parse()` before writing test file |
| FalkorDB socket not running | Health-check on startup; surface a clear error |
| Large repos overflow Gemini context | Cap `affected_functions` to top 20 by `cyclomatic_complexity` before building prompt |
| Line intersection misses on signature lines | Fuzzy fallback: widen search to ±3 lines; log when it fires |
